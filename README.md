# Air GTM Engineering Challenge — Lead Pipeline

A Python pipeline that ingests messy CSV lead data, normalizes it,
enriches each lead via a mocked API, scores leads against a configurable
ruleset, and emits Salesforce-ready outputs — paired with an interactive
Next.js dashboard for real-time ICP scoring and export.

## TL;DR

```bash
# 1. Run the core pipeline (zero dependencies, stdlib only)
py run_pipeline.py

# 2. Start the interactive dashboard
cd frontend
npm install
npm run dev
# open http://localhost:3000
```

## How to run

### Core pipeline (no dependencies)

```bash
# Run the full pipeline with defaults
py run_pipeline.py

# Custom paths / options
py run_pipeline.py \
    --input data/messy_leads.csv \
    --output-dir output \
    --rules config/scoring_rules.json \
    --seed 42 \
    --today 2026-04-15
```

```bash
# Run the unit tests
py -m unittest discover tests
```

### Interactive dashboard (Next.js)

```bash
cd frontend
npm install      # one-time
npm run dev      # starts on http://localhost:3000
```

**Features:**

- **CSV upload** — drag-and-drop any CSV, or use the default `messy_leads.csv`
- **Manual review** — UI loop to approve duplicate merges and restore dropped records before enrichment and scoring
- **10 ICP scoring sliders** — adjust weights in real-time, scores recalculate live
- **Summary Report** — interactive modal to preview and download a Markdown summary of the pipeline run
- **Score rings** — visual SVG indicators per lead, color-coded by tier
- **Filterable/sortable results** — filter by tier (Hot/Warm/Cool/Low), search by name/email/company, sort by score/name/company
- **Lead detail panel** — full enrichment data, product-usage signals, ICP score breakdown with progress bars, data quality issues, pipeline scoring rules
- **Export** — toggle between Salesforce CSV and JSON, preview output, one-click download

The dashboard calls the Python pipeline through Next.js API routes (`child_process.spawn`). The pipeline runs server-side; ICP scoring happens client-side so slider adjustments are instant.

## Pipeline stages

1. **Ingest + normalize** (`src/normalize.py`, `src/pipeline.py`)
   Each field gets its own normalizer. Per-row data quality issues are
   collected in a `data_quality_issues` list.

2. **Drop invalid rows.** Only hard-drop rule: **missing/malformed email**.

3. **Dedupe** (`src/dedupe.py`) by normalized email with most-complete merge.

4. **Enrich** (`src/enrich.py`) via a mocked API with retry + exponential
   backoff. Adds company firmographics + **product-usage signals**
   (pricing page views, signup intent, credit usage, asset uploads,
   team collaboration depth, active channels, creative job postings).

5. **Score** (`src/score.py`) using rules from `config/scoring_rules.json`.

6. **Emit** output files to `output/`.

## ICP Scoring — how the dashboard scores leads

The dashboard uses a **percentage-weight system** with 10 criteria, each
adjustable via a slider (0-50%). Weights should sum to 100%.

| Criteria | What it measures | Data source |
|---|---|---|
| Pricing / Demo Page Visits | Website engagement depth | `pricing_page_views` (mocked) |
| Signup / Payment Intent | Product signup signals | `source` + `has_signup_intent` |
| Free-Credit Usage | Product adoption | `credit_usage_pct` (mocked) |
| Asset Upload Volume | Feature engagement | `asset_uploads` (mocked) |
| Team Collaboration Depth | Viral/expansion signal | `teammate_invites` (mocked) |
| Company Size Fit | ICP firmographic match | `company_size_min` (enrichment) |
| Industry Match | Target market alignment | `industry` (enrichment) |
| Marketing / Creative Team | Buyer persona match | `title` + `seniority_level` |
| Channel Expansion | Growth signal | `active_channels` (mocked) |
| Creative Hiring Signals | Growth intent | `creative_job_postings` (mocked) |

**Tier thresholds:** Hot >= 85, Warm >= 70, Cool >= 55, Low < 55

Each criterion scores 0-100% of its weight based on signal strength
(e.g., 200+ asset uploads = full weight, 50+ = 70%, 10+ = 40%).
Adjusting a slider immediately rescores all leads.

## Assumptions

| Assumption | Rationale |
|---|---|
| **`CA` = Canada**, not California | Column is labeled "Country". |
| **`MM/DD/YYYY` is US format** | US-centric dataset. |
| **7-digit phones are unusable** | Can't dial without area code. |
| **Duplicates merge by email** | Different emails = distinct leads. |
| **Salesforce requires `LastName` and `Company`** | Fallbacks applied. |
| **Product-usage signals are mocked** | Deterministic by email hash. Swap for real product DB in production. |

## Trade-offs

- **Stdlib only** for the pipeline (no pandas). Small dataset; zero install friction.
- **Client-side ICP scoring** in the dashboard. Sliders feel instant; the
  Python pipeline still produces its own scores for the file outputs.
- **Product-usage data is mocked.** Designed so swapping to a real product
  analytics source (Segment, Amplitude, app DB) is a single-function change
  in `enrich.py`.
- **Single-file React component.** Keeps the frontend simple and portable.
  Can be broken into components when the UI grows.

## How I'd scale / productionize this

1. **Orchestration.** Wrap `pipeline.run()` as a Prefect/Airflow task.
2. **Real enrichment.** Swap `_call_mock_api` for Clearbit/ZoomInfo. Swap
   `_synthesize_product_signals` for a Segment/Amplitude query.
3. **State + idempotency.** Persist to Postgres, UPSERT by email.
4. **Salesforce ingestion.** Use SF Bulk API 2.0 with the CSV output.
5. **Observability.** Structured logs + metrics for drop rate, enrichment
   success rate, score distribution.
